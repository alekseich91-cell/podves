# Rigging Load Calculator — Design Spec

**Дата:** 2026-04-24
**Статус:** черновик, ожидает ревью

## 1. Цель

Инструмент для быстрого расчёта нагрузок на точки подвеса плоских грид-конструкций из прямых ферм. Пользователь — практикующий риггер/техдир шоу-ивент индустрии. Основные вопросы, на которые отвечает калькулятор:

- Сколько кг приходится на каждую точку подвеса?
- Какая точка перегружена относительно своего лимита?
- Какой суммарный вес конструкции?
- Что будет в worst-case (если соседняя точка откажет)?

## 2. Область (scope)

### В MVP:
- Плоские 2D-гриды из прямых сегментов ферм (прямоугольники, L, П, кресты, любые связные конфигурации).
- До ~10 точек подвеса на проект (верхний предел не жёсткий, но оптимизировано под типовой шоу-грид).
- Расчёт методом рычага по пролётам + worst-case.
- Визуальный 2D-редактор в плане.
- Свободные позиции точек подвеса и приборов на ферме (не «по сетке»).
- Палитра типов приборов в рамках проекта.
- Автосохранение в localStorage + JSON экспорт/импорт.

### Вне MVP (задел на будущее):
- **Ноги / ground-support.** Возможность «перенести часть грида на пол». Данные-модель поддерживает расширение, но UI и физика в MVP не делаются.
- **Общий каталог оборудования** (моторы/фермы по моделям).
- **PDF-отчёт / печать.**
- **Многопользовательский режим / облачное хранение.**
- **3D-визуализация.**
- **Наклонные фермы и пространственные конструкции.**

## 3. Физика расчёта

### 3.1 Рычажный метод по пролётам

Для каждой фермы (сегмента) длиной L, погонным весом w, с точками подвеса p₁ < p₂ < ... < pₙ (отсортированы по расстоянию от `fromNode`):

- **0 точек подвеса на ферме** → `warning: сегмент без опоры`, расчёт не продолжаем.
- **1 точка** → вся масса фермы + все приборы на ней суммируются в эту точку (`warning: единственная опора на ферме`).
- **2+ точек** → ферма режется на пролёты `[pᵢ, pᵢ₊₁]`. Каждый пролёт — простая балка на двух опорах:
  - Равномерная нагрузка `w·(pᵢ₊₁−pᵢ)` распределяется 50/50 на концы пролёта.
  - Точечные приборы весом `P` на расстоянии `d` от `pᵢ` в пределах пролёта: реакция на `pᵢ` = `P·(s−d)/s`, на `pᵢ₊₁` = `P·d/s`, где `s = pᵢ₊₁−pᵢ`.
  - Консоли (участки `[0, p₁)` и `(pₙ, L]`) — всё (ферма + приборы там) идёт на ближайшую крайнюю точку.
- **Вклады суммируются** по каждой точке подвеса.

### 3.2 Сборка по узлам грида

Точка подвеса с `anchor.kind === "node"` собирает вклады от **всех** ферм, сходящихся в этом узле (для каждой такой фермы она является концевой точкой подвеса на стороне этого узла).

Моторы/лебёдки (`motors[]`) добавляются к реакции своей точки напрямую: `R(hangPoint) += Σ motor.weight`.

### 3.3 Упрощение (осознанная консервативность)

Метод трактует каждую точку подвеса как шарнир. Для замкнутых контуров (прямоугольная рама из 4 ферм на 4 точках) результат чуть консервативнее точного FEM-решения. Это приемлемо для калькулятора безопасности — ошибка в запас.

### 3.4 Worst-case

Для каждой точки подвеса `p`:
```
worst(p) = R(p) + max(R(q)), q ∈ соседи(p)
```

Соседи `p` — точки подвеса, которые делят с `p` пролёт (то есть лежат на той же ферме по соседству, либо в узле, в котором `p` тоже участвует). Интерпретация: «если одна соседняя точка отказала, сколько дополнительно унесёт `p`». Если у `p` нет соседей (единственная точка на ферме), `worst(p) = R(p)`.

Показывается рядом с основным значением (рычаг), шрифтом поменьше.

### 3.5 Проверка лимитов

Каждая точка имеет `maxLoad`. Определяем:
- `ratio = R(p) / maxLoad`
- `< 0.7` → зелёный
- `0.7..1.0` → жёлтый
- `> 1.0` → красный + предупреждение

## 4. Модель данных

```js
Project {
  id: string,              // "prj_<nanoid>"
  name: string,
  createdAt: string,       // ISO 8601
  updatedAt: string,
  schemaVersion: 1,
  grid: Grid
}

Grid {
  nodes: Node[],
  segments: Segment[],
  hangPoints: HangPoint[],
  fixtureTypes: FixtureType[],
  fixtures: Fixture[],
  motors: Motor[]
}

Node {
  id: string,              // "n_<nanoid>"
  position: { x: number, y: number }   // метры
}

Segment {
  id: string,              // "s_<nanoid>"
  fromNodeId: string,
  toNodeId: string,
  weightPerMeter: number,  // кг/м
  note?: string
}

HangPoint {
  id: string,              // "hp_<nanoid>"
  anchor:
    | { kind: "node", nodeId: string }
    | { kind: "segment", segmentId: string, distance: number },  // метры от fromNode
  maxLoad: number,         // кг
  label?: string
}

FixtureType {
  id: string,              // "ft_<nanoid>"
  name: string,            // "Pointe", "Robin MMX", ...
  weight: number           // кг за экземпляр
}

Fixture {
  id: string,              // "fx_<nanoid>"
  typeId: string,          // ссылка на FixtureType
  segmentId: string,
  distance: number         // метры от fromNode
}

Motor {
  id: string,              // "mt_<nanoid>"
  hangPointId: string,
  weight: number,          // кг
  label?: string
}
```

### 4.1 Инварианты

- `segment.fromNodeId !== segment.toNodeId`.
- `hangPoint.anchor`, если `kind==="segment"`: `0 ≤ distance ≤ length(segment)`.
- `fixture.distance`: `0 ≤ distance ≤ length(segment)`.
- Все ID-ссылки разрешимы (нет «висящих» ID).

### 4.2 Каскадное удаление

- Удаление узла → удалить все сегменты, у которых он `from` или `to` → каскад по правилам ниже.
- Удаление сегмента → удалить все точки подвеса с `anchor.kind==="segment"` и этим сегментом → удалить все приборы на нём → у моторов, привязанных к удаляемым точкам, удаление каскадится.
- Удаление точки подвеса → удалить моторы, привязанные к ней.
- Удаление FixtureType → удалить все экземпляры Fixture этого типа.

### 5.2 Принципы декомпозиции

- `physics/*` — чистые функции, без DOM. На вход — `Project`, на выход — `Report`.
- `model/mutations.js` — чистые функции, возвращающие новую версию грида (без in-place мутаций входа).
- `editor/*` — знает про SVG и мышь, не знает про физику.
- `state.js` — единственный источник истины. Подписчики (editor, summary, inspector) перерисовываются при изменении.
- Рендер после каждого изменения: state.set(...) → notify → свежий `compute(project)` → SVG обновляет числа у точек, таблица обновляется.

### 5.3 Формат отчёта


## 6. UI и взаимодействие

### 6.1 Раскладка

Три зоны:
- **Верхний тулбар:** название проекта, кнопки «Новый / Открыть / Сохранить .json / Загрузить .json», индикатор «Суммарный вес: XXX кг» (всегда виден), undo/redo, snap on/off.
- **Левая узкая панель инструментов:** *Выбор*, *+ Ферма*, *+ Точка подвеса*, *+ Прибор*, *+ Лебёдка*, *Удалить*.
- **Центр:** SVG-холст со схемой грида. Сетка в метрах, контролы зума внизу.
- **Правая панель:** инспектор выделенного элемента (контекстно-зависимый) + палитра типов приборов + сводная таблица нагрузок.

### 6.2 Ключевые сценарии

**Создание фермы:**
Режим «+ Ферма» → клик в точке A (создаётся первый узел) → клик в точке B (создаётся второй узел и сегмент между ними). Если второй клик в пределах ~10 px от существующего узла — snap, используется существующий.

**Добавление точки подвеса:**
Режим «+ Точка» → клик по ферме → точка ставится на сегмент на ближайшей к клику позиции, `maxLoad = 500` кг по умолчанию. Если клик близко к узлу — `anchor.kind === "node"`.

**Добавление приборов (штамп):**
В правой панели в блоке «Приборы проекта» добавил тип `{ name: "Pointe", weight: 24 }`. Клик по строке типа → тип активен как штамп (курсор подсвечивает). Клик по ферме → экземпляр ставится в позицию клика. Следующий клик → ещё один, и т.д. ESC или смена режима — штамп выключается.

Редактирование типа (изменил вес с 24 на 25 кг) — все существующие экземпляры мгновенно пересчитываются (они ссылаются на тип).

**Добавление лебёдки:**
Режим «+ Лебёдка» → клик по точке подвеса → появляется мотор с весом по умолчанию (можно поменять в инспекторе). Один мотор на точку (MVP-упрощение; если понадобится несколько — расширим).

**Перемещение:**
- Таскать узел — обе примыкающие фермы обновляют свои концы.
- Таскать точку подвеса — если `anchor.kind==="segment"`, скользит вдоль сегмента; если `"node"` — точка не таскается напрямую (она «живёт в узле»), но едет вместе с узлом при перетаскивании узла.
- Таскать прибор — скользит вдоль своей фермы.

**Snap-to-grid:**
По умолчанию 0.1 м (10 см), применяется при создании и перетаскивании узлов/точек/приборов. Переключатель в тулбаре.

**Zoom/pan:**
- Колёсико мыши — зум к курсору.
- Space + drag или средняя кнопка — панорамирование.
- Кнопка «в размер» в нижнем левом углу — вписать грид в вьюпорт.

**Undo/redo:**
История последних 50 мутаций state. `Ctrl+Z` / `Ctrl+Shift+Z`. История чистится при импорте/новом проекте.

### 6.3 Рендер результатов

Возле каждой точки подвеса на схеме:
```
   ● 185 кг        ← крупно, это лежащий рычаг-расчёт
   (worst 310)     ← помельче, worst-case
```

Цвет круга точки — по `status`:
- `"ok"` (< 70% лимита) — зелёный
- `"warn"` (70–100%) — жёлтый
- `"over"` (> 100%) — красный + мигающий восклицательный знак

В правой панели — таблица:

| Точка | Рычаг | Worst | Лимит | % |
|---|---|---|---|---|
| A1 | 185 | 310 | 500 | 37% |
| A2 | ... | ... | ... | ... |

Под таблицей:
- Ферм: XXX кг
- Приборов: XXX кг
- Лебёдок: XXX кг
- **Итого: XXX кг** (та же цифра, что в шапке тулбара)

Предупреждения (без опор, перегруз) — красной плашкой над холстом.

## 7. Хранение

### 7.1 localStorage

- Ключ `podves.activeProjectId` — ID проекта, открытого в редакторе.
- Ключ `podves.projects.<id>` — полный JSON проекта по его ID (и активный, и все недавние хранятся единообразно).
- Ключ `podves.recentProjects` — массив `{ id, name, updatedAt }` последних 10 проектов, для списка «Недавние» в UI.
- Автосохранение активного проекта: на каждое изменение state → debounce 500 мс → запись в `podves.projects.<activeProjectId>` и обновление его записи в `recentProjects`.

Ограничение localStorage (~5MB на домен) для калькулятора с ~100 объектов на проект не критично: запас в тысячи раз.

### 7.2 JSON экспорт / импорт

**Экспорт:** кнопка «Сохранить .json» → скачивается файл `podves-<name>-<YYYY-MM-DD>.json`:
```json
{
  "schemaVersion": 1,
  "project": { ...Project }
}
```

**Импорт:** кнопка «Загрузить .json» → выбрать файл → парсинг → проверка `schemaVersion` → (в будущем) миграция → валидация инвариантов → замена активного проекта (с подтверждением, если есть несохранённые изменения).

### 7.3 Миграции

На старте — только `schemaVersion: 1`. При изменениях модели добавляется `migrations/v1_to_v2.js` и диспетчер. В коде предусмотрен хук `migrate(data)` в `persistence.js`, пока — no-op.

## 8. Тесты

Запуск: `node --test tests/*.test.js` (без сборки и зависимостей).

### 8.1 Физика (`physics.test.js`)

Известные кейсы, все с ручным расчётом для сверки:

1. Ферма 10 м, 3 кг/м, две точки подвеса по концам, приборов нет → R₁ = R₂ = 15 кг.
2. Та же ферма, прибор 30 кг в центре → R₁ = R₂ = 30 кг.
3. Та же ферма, прибор 30 кг на 2 м от левой точки → R₁ = 39 кг (15 от фермы + 24 от прибора), R₂ = 21 кг (15 от фермы + 6 от прибора). Сумма = 60 кг = вес фермы + вес прибора.
4. Ферма 12 м, 3 точки подвеса (0, 6, 12 м), вес фермы 36 кг → R_центр = 18 кг, R_крайние = 9 кг.
5. Консоль: ферма 10 м, точки в 2 и 8 м, приборов нет → R = 15 кг каждая (симметрия с одинаковыми консолями).
6. Прямоугольный грид 4 × 8 м из 4 ферм (3 кг/м), 4 точки подвеса в углах, один прибор 40 кг по центру одной из длинных ферм → сумма всех R = суммарный вес (инвариант), распределение: точки на стороне с прибором несут больше, симметрия относительно поперечной оси.
7. Т-узел: 3 фермы сходятся в общем узле, в узле точка подвеса → её реакция включает вклады от всех трёх ферм.

### 8.2 Мутации (`mutations.test.js`)

- Каскадное удаление узла удаляет сегменты и всё на них.
- Каскадное удаление сегмента удаляет точки и приборы.
- Удаление FixtureType удаляет его экземпляры.
- Валидация: нельзя создать сегмент с одинаковыми fromNodeId/toNodeId.

### 8.3 Хранение (`persistence.test.js`)

- Экспорт → парсинг → проект идентичен исходному.
- Импорт с неверным `schemaVersion` → ошибка.
- localStorage round-trip.

## 9. Несохранённые решения

- **Лимит `maxLoad` по умолчанию = 500 кг.** Обсуждаемо; можно сделать настраиваемый «дефолт проекта».
- **Worst-case считается как «отказ одного соседа».** Альтернатива — «держит половину всего грида». Если в процессе использования окажется полезнее — добавим переключатель.
- **Один мотор на точку.** Если в реале бывает «в одной точке подвешена пара лебёдок» — снимается тривиально (массив моторов вместо одного).

## 10. Риски и ограничения

- **Замкнутые контуры с ≥4 точками формально статически неопределимы.** Наше упрощение (шарниры в точках) даёт чуть более консервативный результат. Пользователь должен это понимать — в UI стоит одноразовая подсказка «это упрощённый расчёт, не для сертификации».
- **Нет поддержки 3D.** Реальные сложные конструкции с порталами и спайдерами не решатся.
- **Нет ног.** Специально отложено — см. п. 2.

## 11. Метрика успеха

- Пользователь собирает П-грид 8×4 м с 8 точками подвеса и ~20 приборами за < 5 минут.
- Изменение веса типа прибора перекалькулирует нагрузки за < 100 мс.
- Спасает от ошибки: подсвечивает перегруженную точку, прежде чем рама улетит на площадку.
